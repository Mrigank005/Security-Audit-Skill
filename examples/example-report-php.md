🔴 CRITICAL — SQL Injection
Confidence: HIGH
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📍 Location: api.php, Line 22

🔍 Vulnerable Code:
```php
$id = $_GET['id'];
$result = $conn->query("SELECT * FROM users WHERE id = " . $id);
```

⚠️ Risk:
  The `id` parameter from `$_GET` is concatenated directly into a SQL query
  without sanitization or parameterization. An attacker can inject arbitrary
  SQL commands.

  Example: `api.php?action=get_user&id=1 OR 1=1--`
  Example: `api.php?action=get_user&id=1 UNION SELECT password,2,3,4 FROM users--`

✅ Recommended Fix:
  Use prepared statements:
```php
$stmt = $conn->prepare("SELECT * FROM users WHERE id = ?");
$stmt->bind_param("i", $_GET['id']);
$stmt->execute();
$result = $stmt->get_result();
```

🔍 Vulnerable Code:

## Findings — File Inclusion & Deserialization

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🟠 HIGH — Local File Inclusion (LFI)
Confidence: HIGH
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📍 Location: index.php, Line 13

🔍 Vulnerable Code:
```php
include($_GET['page'] . ".php");
```

⚠️ Risk:
  The `page` parameter from `$_GET` is used directly in an `include()` call.
  An attacker can include arbitrary PHP files on the server:

  Example: `index.php?page=../../../etc/passwd%00` (null byte in older PHP)
  Example: `index.php?page=../config` (reads config.php credentials)
  With `allow_url_include=On`, this becomes Remote File Inclusion (RFI).

✅ Recommended Fix:
  Use an allowlist of permitted pages:
```php
$allowed = ['home', 'about', 'contact'];
$page = isset($_GET['page']) ? $_GET['page'] : 'home';
if (!in_array($page, $allowed, true)) {
    http_response_code(404);
    exit('Page not found');
}
include($page . ".php");
```

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📍 Location: api.php, Line 51

🔍 Vulnerable Code:
```php
$obj = unserialize($_GET['data']);
```

⚠️ Risk:
  `unserialize()` with user-controlled input can lead to PHP Object Injection.
  If the application (or any loaded library) has classes with dangerous magic
  methods (`__destruct`, `__wakeup`, `__toString`), an attacker can chain them
  to achieve Remote Code Execution.

✅ Recommended Fix:
  Use `json_decode()` instead of `unserialize()` for user data:
```php
$obj = json_decode($_GET['data'], true);
if ($obj === null) {
    http_response_code(400);
    exit('Invalid data');
}
```
