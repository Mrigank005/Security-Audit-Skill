📍 Location: app/routes/auth.py, Line 12

🔍 Vulnerable Code:
```python
SECRET_KEY = 'dev-secret'
```

⚠️ Risk:
  The Flask `SECRET_KEY` is used to sign session cookies. With this secret
  exposed, an attacker can forge session cookies, impersonate any user, and
  bypass authentication entirely. The value `dev-secret` is trivially guessable.

✅ Recommended Fix:
```python
import os
SECRET_KEY = os.environ.get('FLASK_SECRET_KEY')
if not SECRET_KEY:
    raise RuntimeError('FLASK_SECRET_KEY environment variable is required')
```

⚠️ Risk:
  The `user_id` parameter from the URL is interpolated directly into a raw
  SQL query using an f-string. An attacker can inject arbitrary SQL:

✅ Recommended Fix:


🔴 CRITICAL — Server-Side Template Injection (SSTI)
Confidence: HIGH
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📍 Location: app/utils/render.py, Lines 17–19

🔍 Vulnerable Code:
```python
template = request.form.get('template', '')
name = request.form.get('name', 'Guest')
return render_template_string(template, name=name)
```

⚠️ Risk:
  User-supplied input is passed directly to `render_template_string()` as the
  template itself. An attacker can inject Jinja2 template expressions to:
  - Read application config: `{{ config.items() }}`
  - Execute arbitrary Python code: `{{ ''.__class__.__mro__[1].__subclasses__() }}`
  - Read server files, establish reverse shells, etc.

  This is effectively Remote Code Execution (RCE).

✅ Recommended Fix:
  Never pass user input as the template. Use a fixed template with variables:
```python
template = "<p>Hello {{ name }}!</p>"
return render_template_string(template, name=name)
```

📍 Location: app/utils/render.py, Lines 27–29

🔍 Vulnerable Code:
```python
name = request.args.get('name', 'World')
html = f"<h1>Hello {name}!</h1><p>Welcome to our platform.</p>"
return render_template_string(html)
```

⚠️ Risk:
  User input is injected into a string via f-string, which is then passed to
  `render_template_string()`. The f-string produces a template that contains
  the raw user input, so Jinja2 template expressions in `name` will execute.

  Example: `GET /api/preview?name={{config.SECRET_KEY}}`

✅ Recommended Fix:
```python
return render_template_string(
    "<h1>Hello {{ name }}!</h1><p>Welcome to our platform.</p>",
    name=name
)
```

📍 Location: app/routes/users.py, Lines 64–69

🔍 Vulnerable Code:
```python
filename = request.args.get('name', '')
filepath = os.path.join('/var/uploads', filename)

if not os.path.exists(filepath):
    abort(404)

return send_file(filepath)
```

⚠️ Risk:
  The filename comes directly from the query parameter with no sanitization.
  An attacker can traverse the directory structure:

  Example: `GET /api/files/download?name=../../../etc/passwd`

  This exposes any file on the server readable by the application process.

✅ Recommended Fix:
```python
filename = os.path.basename(request.args.get('name', ''))
filepath = os.path.realpath(os.path.join('/var/uploads', filename))

if not filepath.startswith('/var/uploads/'):
    abort(400, 'Invalid filename')

if not os.path.exists(filepath):
    abort(404)

return send_file(filepath)
```

Confidence: HIGH
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📍 Location: app/routes/auth.py, Lines 28, 53

🔍 Vulnerable Code:
```python
password_hash = hashlib.md5(password.encode()).hexdigest()
```

⚠️ Risk:
  MD5 is cryptographically broken. It is extremely fast to compute, making
  brute-force and rainbow table attacks trivial. An attacker who obtains the
  database can crack most passwords in seconds.

✅ Recommended Fix:
```python
import bcrypt

# Hashing
password_hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))

# Verifying
is_valid = bcrypt.checkpw(password.encode(), stored_hash)
```

📍 Location: app/__init__.py, Line 19

🔍 Vulnerable Code:
```python
app.run(debug=True, host='0.0.0.0', port=5000)
```

⚠️ Risk:
  Flask's debug mode exposes the interactive Werkzeug debugger. If the
  application crashes, an attacker can execute arbitrary Python code through
  the debugger console in the browser. Additionally, `host='0.0.0.0'` exposes
  the application on all network interfaces.

✅ Recommended Fix:
```python
import os
app.run(
    debug=os.environ.get('FLASK_DEBUG', 'false').lower() == 'true',
    host='127.0.0.1',
    port=int(os.environ.get('PORT', 5000))
)
```
